# AGENTS 协作规范

## 项目概况
- 项目类型：本地优先的 Electron + 静态前端密码管理器。
- 核心入口：`index.html`、`styles.css`、`app.js`、`electron/main.cjs`、`electron/preload.cjs`。
- 数据模式：浏览器本地存储 + WebCrypto AES-GCM/PBKDF2，本项目不依赖后端服务。

## Token 节省规则
- 默认只读取与任务直接相关的文件，避免全项目扫描。
- 除非任务明确需要，不读取以下目录或文件：
  - `node_modules/`
  - `dist/`
  - `.git/`
  - `.idea/`
  - `screenshots/`
  - `package-lock.json`
- 修改 UI 优先查看 `index.html`、`styles.css`，修改业务逻辑优先查看 `app.js`。
- 修改桌面端能力优先查看 `electron/main.cjs`、`electron/preload.cjs`。
- 不重复总结已知项目背景，直接围绕当前需求处理。

## 工作目标
- 以当前需求为中心，优先交付可运行、可验证的结果。
- 优先最小改动，避免无关重构。
- 不修改与任务无关文件。

## 实施原则
- 先读代码再改代码，遵循现有项目风格。
- 不确定且高风险时先澄清；低风险场景做合理假设并推进。
- 优先复用现有能力，谨慎新增依赖。
- 禁止为了“顺手整理”而格式化代码。
- 禁止删除、改写、搬动原有注释，除非对应逻辑已被实际修改且注释失真。
- 每次更改代码都要写明必要注释；已有清晰注释时不强行新增噪音注释。

## 安全与边界
- 未明确授权不执行破坏性命令，例如强制重置、批量删除、清空数据。
- 不提交密钥、令牌、隐私数据。
- 涉及密码、主密码、导入导出、加密、Electron IPC 时优先考虑安全边界。
- 外部输入必须校验，错误路径要显式处理。

## 验证方式
- Java 程序不用编译测试；本项目不是 Java 项目。
- 前端静态逻辑改动优先做语法级检查和浏览器手动验证。
- Electron 主进程或 preload 改动后，优先说明需要重启桌面应用验证。
- 改依赖或打包配置时再考虑运行 `npm` 相关命令。
- 不因验证而读取或改动 `dist/` 产物。

## 常用命令
```bash
npm run dev
npm run desktop
npm run dist:mac
npm run dist:win
```

## 提交约定
- Commit 推荐格式：`feat(scope): summary`、`fix(scope): summary`。
- PR 需写明：问题、方案、影响、验证步骤。

## 沟通风格
- 简洁直接，先给结论，再解释原因。
- 能用表格就用表格，能给步骤就给步骤。
- 遇到风险点直接提醒。
- 不确定的地方明确说明，不胡编。
